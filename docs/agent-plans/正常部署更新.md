# 正常部署更新

适用于教练助手服务器的日常代码更新。默认环境：

- GitHub 更新分支：`codex/login-history`
- 服务器项目目录：`/opt/apps/teacher`
- 服务器部署分支：`server-login-history`
- systemd 服务：`teacher`
- Node.js：`/root/.nvm/versions/node/v20.20.2`
- Node 直连端口：`4173`
- Nginx 对外端口：`4175`
- SQLite 数据库：`/opt/apps/teacher-data/teacher.sqlite`

> 本流程不调用真实 DeepSeek API。代码应先在本地通过全部测试并推送到 GitHub，再更新服务器。

> 日常更新通常只执行第二节，不需要每次都运行 `npm ci`、重编译 `sqlite3` 或迁移数据库。

## 一、更新前检查

登录服务器后逐段执行：

```bash
cd /opt/apps/teacher
```

```bash
git status --short --branch
git branch --show-current
systemctl is-active teacher
```

预期：

- 当前分支是 `server-login-history`。
- tracked files（已跟踪文件）没有本地修改。
- 服务状态是 `active`。

获取 GitHub 最新代码：

```bash
git fetch origin codex/login-history
```

查看将要更新的提交：

```bash
git log --oneline HEAD..origin/codex/login-history
```

检查是否涉及依赖或数据库迁移：

```bash
git diff --name-only HEAD origin/codex/login-history -- package.json package-lock.json server/database scripts/migrate.js
```

- 没有任何输出：执行第二节“日常快速更新”。
- 出现文件路径：执行第三节“完整更新”。
- 如果 `git log` 没有输出，说明服务器已经是最新版本，不需要更新。

## 二、日常快速更新

适用于只修改 HTML、CSS、JavaScript、提示词、测试或文档，并且上一节的依赖与迁移检查没有输出的情况。

记录旧提交并创建回退分支：

```bash
PREVIOUS_COMMIT=$(git rev-parse HEAD)
ROLLBACK_BRANCH="rollback-teacher-$(date +%Y%m%d-%H%M%S)"
git branch "$ROLLBACK_BRANCH" "$PREVIOUS_COMMIT"
```

更新代码：

```bash
git merge --ff-only origin/codex/login-history
```

重启服务：

```bash
systemctl restart teacher
```

验证服务：

```bash
curl -fsS http://127.0.0.1:4173/api/health
curl -fsS http://127.0.0.1:4175/api/health
```

```bash
systemctl is-active teacher
systemctl show teacher -p NRestarts --no-pager
git status --short --branch
```

成功标准：

- 两个健康检查都返回 `{"ok":true}`。
- 服务状态是 `active`。
- Git 工作区没有 tracked files 修改。
- 浏览器可以正常打开 `http://8.163.71.207:4175`。

服务刚重启时，如果第一次健康检查出现连接失败，等待两秒后重新执行该条 `curl` 即可。

## 三、完整更新

以下任一情况必须使用本节：

- `package.json` 或 `package-lock.json` 发生变化。
- `server/database/` 或 `scripts/migrate.js` 发生变化。
- `node_modules` 缺失。
- Node.js 版本发生变化。
- `require('sqlite3')` 加载失败。

### 3.1 记录回退点

```bash
cd /opt/apps/teacher
```

```bash
PREVIOUS_COMMIT=$(git rev-parse HEAD)
TIMESTAMP=$(date +%Y%m%d-%H%M%S)
ROLLBACK_BRANCH="rollback-teacher-$TIMESTAMP"
git branch "$ROLLBACK_BRANCH" "$PREVIOUS_COMMIT"
```

备份 `.env`，但不要输出其中的内容：

```bash
install -d -m 700 /var/backups/teacher
install -m 600 .env "/var/backups/teacher/env.$TIMESTAMP"
```

### 3.2 停止服务并备份数据库

```bash
systemctl stop teacher
```

```bash
cp -a /opt/apps/teacher-data/teacher.sqlite "/var/backups/teacher/teacher.$TIMESTAMP.sqlite"
chmod 600 "/var/backups/teacher/teacher.$TIMESTAMP.sqlite"
```

### 3.3 更新代码和依赖

```bash
git merge --ff-only origin/codex/login-history
```

启用服务器的 Node.js 20：

```bash
export NVM_DIR=/root/.nvm
source "$NVM_DIR/nvm.sh"
nvm use 20.20.2
NODE_HOME=/root/.nvm/versions/node/v20.20.2
```

如果依赖文件发生变化或 `node_modules` 缺失，按以下步骤处理。

如果现有 `node_modules` 目录存在，先保留它作为依赖回退副本：

```bash
mv node_modules "/opt/apps/teacher-node_modules-$TIMESTAMP"
```

然后重新安装依赖：

```bash
npm ci --registry=https://registry.npmmirror.com --no-audit --no-fund
```

Ubuntu 22.04 的 `glibc 2.35` 不能直接使用当前 `sqlite3` 下载到的部分预编译文件。只要执行过 `npm ci`，就必须使用服务器已有的 Node headers（头文件）从源码重编译：

```bash
npm_config_build_from_source=true \
npm_config_nodedir="$NODE_HOME" \
MAKEFLAGS=-j1 \
npm rebuild sqlite3 --foreground-scripts --loglevel=info
```

验证 `sqlite3`：

```bash
node -e "require('sqlite3'); console.log('sqlite3 load ok')"
```

```bash
ldd node_modules/sqlite3/build/Release/node_sqlite3.node
```

必须满足：

- 出现 `sqlite3 load ok`。
- `ldd` 没有 `not found`。
- 不出现 `GLIBC_2.38 not found`。

### 3.4 迁移和服务器测试

加载服务器 `.env` 执行 migration（迁移）：

```bash
node --env-file=/opt/apps/teacher/.env scripts/migrate.js
```

预期输出：

```text
Database migrations completed.
```

限制并发运行服务端测试，避免服务器内存压力过大：

```bash
npm run test:server -- --test-concurrency=1
```

只有测试全部通过，才继续启动服务。

### 3.5 启动并验证

```bash
systemctl start teacher
```

```bash
curl -fsS http://127.0.0.1:4173/api/health
curl -fsS http://127.0.0.1:4175/api/health
```

```bash
systemctl is-active teacher
systemctl show teacher -p NRestarts --no-pager
git status --short --branch
```

最后使用浏览器完成一次不调用 DeepSeek 的检查：

- 首页可以打开。
- 注册、登录和退出可以正常使用。
- 历史记录页面可以打开。
- 未登录访问受保护接口仍返回 `401`。

## 四、失败时回退

如果新版本启动失败，先查看日志：

```bash
systemctl status teacher --no-pager -l
journalctl -u teacher -n 100 --no-pager
```

代码回退采用刚才创建的回退分支，不使用 `git reset --hard`：

```bash
systemctl stop teacher
git switch "$ROLLBACK_BRANCH"
```

如果本次更新执行过 `npm ci`，同时恢复更新前的依赖：

```bash
mv node_modules "/opt/apps/teacher-node_modules-failed-$TIMESTAMP"
mv "/opt/apps/teacher-node_modules-$TIMESTAMP" node_modules
```

没有执行过 `npm ci` 时，跳过以上两条 `mv`。然后启动旧版本：

```bash
systemctl start teacher
```

然后验证：

```bash
curl -fsS http://127.0.0.1:4173/api/health
curl -fsS http://127.0.0.1:4175/api/health
```

说明：

- 快速更新失败时，切回回退分支即可恢复旧代码。
- 完整更新若在 migration 后失败，不要直接覆盖数据库；先保留日志和 `/var/backups/teacher/teacher.$TIMESTAMP.sqlite`。
- 数据库恢复属于破坏性操作，必须确认恢复目标和备份文件后单独执行。
- 回退成功后，服务器会临时停留在 `rollback-teacher-*` 分支。下次部署前应先确认问题已修复，再切回 `server-login-history`。

## 五、服务器特殊注意事项

- 继续使用 HTTP，因此 `.env` 中应保持 `SESSION_COOKIE_SECURE=false`。
- `.env` 中不要保留空行；该服务器的 Node.js `--env-file` 曾出现空行后首个变量未加载的问题。
- 不要显示、复制或提交 `SESSION_SECRET`、`DEEPSEEK_API_KEY` 等真实值。
- 不要提交 `/opt/apps/teacher-data/teacher.sqlite`、日志、Cookie 或 Session 数据。
- Nginx 的 `4175` 配置必须保留 `proxy_set_header Host $http_host;`，否则同源校验会丢失端口并返回 `AUTH_CSRF_INVALID`。
- 官方 `registry.npmjs.org` 在该服务器上曾连接超时；依赖安装使用 `https://registry.npmmirror.com`。
- 服务器没有 swap（交换空间）且可用内存有限，`sqlite3` 编译使用 `MAKEFLAGS=-j1`。
- 不要删除旧回退分支、旧数据库备份或旧 `node_modules` 备份，除非已确认新版本稳定且再次获得删除授权。

## 六、日常更新最短版

确认第一节的依赖与迁移检查没有输出后，日常实际只需要：

```bash
cd /opt/apps/teacher
git fetch origin codex/login-history
git branch "rollback-teacher-$(date +%Y%m%d-%H%M%S)" HEAD
git merge --ff-only origin/codex/login-history
systemctl restart teacher
curl -fsS http://127.0.0.1:4173/api/health
curl -fsS http://127.0.0.1:4175/api/health
```

这就是普通代码更新的主流程；第三节只在依赖、数据库或运行环境发生变化时使用。
